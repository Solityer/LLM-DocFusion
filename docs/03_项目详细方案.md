## 1. 项目背景与问题定义

### 1.1 业务背景

在政务、科研、金融、医疗等行业，存在大量"数据汇总报表"类工作：

- 从多个部门、多个系统导出的文件（Excel、Word、文本报告等）中提取关键数据指标；
- 将提取结果按照规定格式填入标准化汇报模板；
- 汇总完成后需人工核查，防止填写错误。

这一流程存在三个核心问题：

**数据源异构性**
- 同一指标可能出现在 Excel 表格、Word 正文、TXT 报告等不同格式的不同位置；
- 不同文件对同一概念的表述方式各异（如"GDP"、"地区生产总值"、"产值"）；
- 格式归一化困难，传统工具无法统一处理。

**语义匹配难度**
- 模板表头与数据源字段往往不完全相同（存在同义词、缩写、全称等变体）；
- 需要理解上下文语义才能正确对应数据字段；
- 传统规则匹配在面对自然语言描述时覆盖率不足。

**人工成本高**
- 一份覆盖 100 个城市的经济数据报表，手动填写需 2～4 小时；
- 人工差错率约 3%～8%，需要二次校验；
- 重复性工作耗费大量高素质劳动力，ROI 极低。

### 1.2 需解决的核心技术挑战

| 挑战 | 难点描述 | 本项目解决方案 |
|:----:|:--------:|:-------------:|
| 多格式统一读取 | 5种不同格式，结构迥异 | DocumentBundle 中间层统一表示 |
| 语义字段对齐 | 同义词、缩写、上下文歧义 | LLM 语义理解辅助 |
| 多实体批量提取 | 百强榜100条数据，覆盖率挑战 | 分块迭代 LLM 调用 |
| 格式保持写回 | 填充后需保留原始样式 | openpyxl/python-docx 精准写回 |
| 结果可信度 | 如何衡量填充质量 | 置信度评分 + 证据链追溯 |

## 2. 总体设计思路

### 2.1 设计哲学

**"规则优先，LLM 备用"**：确定性强的场景（精确数字匹配、标准表头对齐）优先用规则处理，确保速度和稳定性；不确定场景（自然语言语义理解、跨段落信息整合）才调用 LLM，控制 API 开销。

**"统一中间层，解耦上下游"**：引入 DocumentBundle 中间表示，将"读取各种格式"与"处理文档内容"完全解耦，使得新增文件格式支持时无需修改任何下游代码。

**"流水线架构，阶段可观测"**：处理流程设计为线性流水线，每个阶段独立、可测试，且向调用方推送实时进度，便于调试和用户体验优化。

### 2.2 技术路线选择

| 方向 | 选型 | 理由 |
|:----:|:----:|:----:|
| LLM 推理 | Ollama + qwen2.5:14b | 本地化部署，无 API 费用，中文能力强 |
| Web 框架 | FastAPI | 原生 async，适合文件上传+长任务 |
| 文档处理 | openpyxl + python-docx | 成熟稳定，格式保持能力强 |
| 数据验证 | Pydantic v2 | 类型安全，自动序列化 |
| 前端 | 原生 HTML/JS | 轻量，无需构建工具，部署简单 |

## 4. 核心数据模型设计

### 4.1 文档统一中间表示

```python
class DocumentBundle(BaseModel):
    document_id: str          # 文档唯一标识
    source_file: str          # 原始文件路径
    file_type: str            # 文件类型：excel/word/markdown/text/csv
    role: FileRole            # 角色：source（数据源）/ template（模板）
    text_blocks: list[TextBlock]     # 文本块列表（段落/标题）
    tables: list[NormalizedTable]    # 归一化表格列表
    raw_text: str             # 全文原始文本（供 LLM 上下文使用）
    metadata: dict            # 扩展元数据（标题、段落数、表格数等）
```

**设计要点**：
- `text_blocks` 和 `tables` 分别存储非结构化和结构化内容，检索时分别处理
- `raw_text` 保存完整文本用于 LLM 大上下文推理
- `role` 字段标记该文档是作为"数据源"还是"模板"，影响后续处理逻辑

### 4.2 模板结构描述

```python
class TemplateSchema(BaseModel):
    template_id: str              # 模板唯一标识
    source_file: str              # 模板文件路径
    file_type: str                # 模板格式
    structure_type: StructureType # tabular / text / mixed
    fields: list[TemplateField]   # 需填充的字段列表
    tables: list[TemplateTable]   # 需填充的表格列表
    sections: list[TemplateSection] # 需填充的文本段落列表
```

### 4.3 需求规格

```python
class RequirementSpec(BaseModel):
    raw_text: str           # 原始需求文本
    time_range: dict        # 时间约束 {start, end}
    entity_keywords: list   # 实体关键词（地区、机构等）
    filter_conditions: dict # 过滤条件
    focus_fields: list      # 重点关注字段
```

### 4.4 处理响应

```python
class ProcessResponse(BaseModel):
    task_id: str              # 任务 ID
    status: str               # processing / completed / failed
    current_stage: str        # 当前处理阶段
    progress: float           # 进度 0.0 ~ 1.0
    result: FilledResult      # 填充结果详情
    model_usage: ModelUsageSummary  # LLM 调用统计
    template_statuses: list   # 各模板处理状态
```

## 5. 各模块详细设计

### 5.1 文档解析服务

**职责**：读取任意格式文件，归一化为 DocumentBundle。

**支持格式与解析策略**：

| 格式 | 解析库 | 特殊处理 |
|:----:|:------:|:-------:|
| Excel (.xlsx/.xls) | openpyxl + pandas | 自动识别表头行（扫描前5行），多 Sheet 均解析 |
| Word (.docx) | python-docx | 段落+表格双通道，保留标题层级 |
| Markdown (.md) | 自研解析器 | 识别 ATX/Setext 标题，提取 GFM 表格 |
| TXT (.txt) | 规则分块 | 按空行、编号、句号切分文本块 |
| CSV (.csv) | pandas | 自动推断分隔符，处理 BOM 编码 |

**核心算法——表头行识别**：
```python
# 扫描前5行，找到非空单元格数量 >= 2 的第一行作为表头
for i in range(min(5, len(df))):
    non_empty = df.iloc[i].dropna()
    if len(non_empty) >= 2:
        header_row_idx = i
        break
```

### 5.2 模板解析服务

**职责**：解析模板文件，识别需要填充的区域、字段和占位符。

**占位符识别规则**（多种格式兼容）：
```
{{字段名}}    →  双大括号格式
{字段名}      →  单大括号格式
【字段名】    →  中文书名号格式
```

**表格可写区域推断**：
- 识别表头行（至少包含一个有意义的中文/英文标题）
- 表头行以下所有空行标记为可写行
- 对半填充模板（部分已有数据）：识别空单元格所在列

### 5.3 需求解析服务

**职责**：将用户自然语言需求转换为结构化 RequirementSpec。

**自动推断机制**：当用户未提供需求时，系统按以下优先级自动推断：
1. 查找模板目录下的 `用户要求.txt` 或 `requirement.txt`；
2. 根据模板文件名和数据源文件名推断（如"2025年城市GDP排名.xlsx"→ 推断关注年份=2025，主题=城市GDP）；
3. 基于模板表头关键词构造最小需求规格。

**结构化解析**：
- 时间范围：使用正则识别年份、月份、季度表达式
- 实体关键词：提取地名后缀（市/省/区/县）、机构名、产品名
- 过滤条件：识别数值比较表达式（"前100名"、"超过X万"等）

### 5.4 证据检索服务

**职责**：从所有 DocumentBundle 中，检索与当前模板列最相关的文本块和候选表格。

**检索策略**：

**文本块检索**：
- 关键词命中：模板表头词汇、需求关键词出现在 text_block.content 中
- 标题优先：heading_level > 0 的文本块权重提升
- 上下文窗口：返回命中块及其前后相邻块

**表格候选检索**：
- 表头对齐度：计算模板表头与源表格表头的字符串相似度（使用 SequenceMatcher）
- 语义相似度：基于关键词重叠率的轻量语义匹配
- 候选排序：按综合相似度分数排序，取 Top-K 候选

**相似度计算**：
```python
def similarity(a: str, b: str) -> float:
    """字符串相似度：基于最长公共子序列 + 字符重叠"""
    from difflib import SequenceMatcher
    return SequenceMatcher(None, a.lower(), b.lower()).ratio()
```

### 5.5 数据抽取服务

**职责**：从检索结果中精确提取结构化数据记录，对应模板各列。

**混合抽取策略**（三层降级）：

**第一层：规则匹配**
- 数值抽取：正则 `[-+]?\d[\d,]*(?:\.\d+)?` 匹配数字
- 日期抽取：多种日期格式正则组合
- 实体列匹配：检测地名后缀自动对齐实体列
- 置信度：`HIGH`（当规则命中率 > 80%）

**第二层：LLM 辅助抽取**
- 触发条件：规则覆盖率不足，或存在语义歧义
- Prompt 设计：提供表头列表 + 候选文本 → 要求 LLM 返回结构化 JSON
- 模型：qwen2.5:14b，temperature=0.3（低温度保证稳定性）
- 置信度：`MEDIUM`

**第三层：分块迭代提取**
- 触发条件：检测到多实体信号（"百强"、"排名"、"名录"等关键词）
- 策略：将文本按 12 个 block 为一组分块，每块独立 LLM 调用
- 合并：对所有 chunk 结果合并去重（基于主键列去重）
- 置信度：`MEDIUM`（带去重验证）

**多实体信号检测**：
```python
MULTI_ENTITY_SIGNALS = [
    "百强", "排名", "排行", "榜单", "名录", "名单", "各城市", "各省",
    "Top", "ranking", "排行榜", "全景报告", "前100", "前50",
]
```

### 5.6 模板填充服务

**职责**：将抽取的结构化数据按行写入模板，保留原始格式。

**分格式填充策略**：

| 格式 | 技术方案 | 特殊处理 |
|:----:|:--------:|:-------:|
| Excel | openpyxl 精准单元格写入 | 保留样式、字体、边框、合并单元格 |
| Word | python-docx 表格行填充 | 保留段落样式，支持表格行复制 |
| Markdown | 字符串模板替换 | 按 GFM 表格格式重新生成 |
| TXT | 占位符字符串替换 | {{字段名}} 模式原地替换 |
| CSV | csv.writer 逐行写入 | 保留原始列顺序 |

**置信度分级**：
```
rule_exact    →  95%+   （规则精确匹配）
rule_contains →  80%+   （规则包含匹配）
llm_high      →  75%+   （LLM 高置信响应）
llm_medium    →  60%+   （LLM 中置信响应）
llm_inferred  →  40%+   （LLM 推断）
not_found     →  0%     （未找到数据）
```

### 5.7 结果验证服务

**职责**：对填充结果进行多维度质量验证。

**验证维度**：

1. **填充完整性**：已填充字段数 / 总字段数，低于阈值触发告警
2. **数值合理性**：对数值型字段自动检测异常值（超出合理范围、非数值格式）
3. **证据链完整性**：每个填充字段必须有可追溯的来源（source_file, source_table, source_row）
4. **日期格式一致性**：检测填充日期是否符合模板预设格式

### 5.8 流水线编排服务（pipeline_service）

**职责**：编排上述所有服务，管理整体处理流程，推送实时进度。

**进度节点设计**：
```
queued        0%
parsing       10%
template      25%
requirement   35%
retrieval     50%
extraction    65%
filling       80%
validation    90%
completed     100%
```

**异步任务管理**：
- 采用 Python asyncio + threading 混合模型
- 任务状态存储：内存 dict（task_store）
- 进度推送：HTTP 轮询（`/api/status/{task_id}`）

## 6. 处理流程详解

### 6.1 端到端完整流程

### 6.2 多模板批量处理

当用户上传多个模板文件时，系统对每个模板独立执行 5.4～5.7 的流程，共享同一组数据源 DocumentBundle，最终聚合所有模板的填充结果统一返回。

### 6.3 错误处理与降级策略

| 错误场景 | 处理策略 |
|:--------:|:--------:|
| LLM 服务不可用 | 降级为纯规则抽取，结果标注"LLM不可用" |
| 数据源无匹配数据 | 填入空值，记录 low_confidence 告警 |
| 模板格式不支持 | 返回明确错误码，提示支持的格式 |
| 文件读取失败 | 跳过该文件，继续处理其他文件 |
| LLM 返回非法 JSON | json_repair 模块自动修复 |

## 7. 大语言模型集成方案

### 7.1 模型选型

**选择 qwen2.5:14b 的理由**：
- **中文理解能力**：专为中文优化，对政务报告、统计公报等中文文档理解能力显著强于同参数量英文模型
- **指令遵循能力**：严格按 JSON schema 输出结构化结果
- **本地化部署**：通过 Ollama 完全本地化，无需 API 密钥，无数据出境风险
- **资源消耗**：14B 参数在消费级 GPU（如 RTX 3090）或高性能 CPU 上均可运行

### 7.3 Prompt 工程设计

**数据抽取 Prompt 模板**：

## 12. 安全设计

### 12.1 文件上传安全

- **白名单格式验证**：只接受 `.xlsx/.xls/.docx/.md/.txt/.csv`，拒绝可执行文件
- **文件名净化**：使用 `Path(filename).name` 去除路径遍历字符
- **文件大小限制**：默认单文件上传上限 50MB
- **存储隔离**：每个任务单独子目录（`uploads/{uuid}/`），任务完成后自动清理

### 12.2 数据隐私

- **完全本地化**：LLM 推理通过本地 Ollama 服务，数据不出服务器
- **临时文件清理**：任务完成后自动删除上传文件（`cleanup_after_run`）
- **无持久化用户数据**：系统不存储任何用户个人信息

### 12.3 接口安全

- **输入验证**：所有 API 参数通过 Pydantic 模型严格验证
- **路径遍历防护**：下载接口只允许访问 `outputs/` 目录内的文件
- **错误信息脱敏**：生产环境不暴露内部堆栈信息
