# 关键模块概要设计与创新要点说明

> 说明：本文档重点阐述**不超过3个**关键创新点，并对每个创新点进行概要设计说明。

## 创新点一：DocumentBundle——多格式文档统一中间表示层

### 1.1 创新背景与动机

传统文档处理工具通常针对单一格式开发（如专门的 Excel 处理库、Word 处理库），导致面对多格式输入时需要维护多套处理逻辑，系统耦合度高、扩展困难。

本项目提出 **DocumentBundle** 统一中间表示层，作为原始文档格式与下游业务逻辑之间的"翻译层"，实现真正的格式无感知处理。

### 1.2 设计方案

**数据模型设计**：

```python
class DocumentBundle(BaseModel):
    """
    统一文档中间表示。
    无论原始格式（Excel/Word/Markdown/TXT/CSV），
    均被归一化为此结构，供后续所有模块使用。
    """
    document_id: str               # 文档唯一标识（取自文件名 stem）
    source_file: str               # 原始文件路径（保留溯源信息）
    file_type: str                 # 格式类型：excel/word/markdown/text/csv
    role: FileRole                 # 角色：source（数据源）/ template（模板）
    text_blocks: list[TextBlock]   # 文本块列表——非结构化内容
    tables: list[NormalizedTable]  # 归一化表格列表——结构化内容
    raw_text: str                  # 完整原始文本（LLM 大上下文推理用）
    metadata: dict                 # 扩展元数据（标题、段落数等）

class TextBlock(BaseModel):
    content: str       # 文本块内容
    heading_level: int # 0=正文段落；1-6=对应 Markdown/Word 标题层级
    block_index: int   # 在文档中的顺序索引

class NormalizedTable(BaseModel):
    table_index: int        # 表格序号
    sheet_name: str         # 所属 Sheet（Excel）/ 表格标题
    headers: list[str]      # 归一化后的表头列表
    rows: list[list[str]]   # 数据行，每行为字符串列表
    row_count: int
    col_count: int
```

**各格式解析策略**：

| 格式 | 文本块提取策略 | 表格提取策略 |
|:----:|:------------:|:-----------:|
| Excel | 非表格单元格内容聚合为文本块 | 自动识别表头行（扫描前5行，非空列 ≥ 2） |
| Word | 按段落（paragraph）切分，保留标题层级 | 直接映射 `docx.Table` → `NormalizedTable` |
| Markdown | 按标题层级切分段落，保留 ATX 层级 | 提取 GFM 管道表格（`\|` 分隔行）|
| TXT | 按双换行符、编号前缀切分文本块 | 尝试识别固定宽度/制表符分隔表格 |
| CSV | 整体转为单一文本摘要 | 直接作为单张 `NormalizedTable` |

**架构解耦效果**：

```
[Excel 解析] ──┐
[Word 解析]  ──┤
[MD 解析]    ──┼──→  DocumentBundle  ──→  [检索] → [抽取] → [填充]
[TXT 解析]   ──┤         (统一接口)
[CSV 解析]   ──┘
    ↑
  新增格式只需在此处添加解析器，
  下游所有模块零改动
```

### 1.3 创新价值

1. **扩展性**：新增文件格式支持只需实现一个 `_read_XXX(bundle, path)` 函数，下游 7 个处理模块完全无需修改；
2. **双通道设计**：`text_blocks`（非结构化）和 `tables`（结构化）并行存储，分别服务于不同的检索和抽取策略；
3. **原文保留**：`raw_text` 字段保存完整原始文本，当结构化抽取失败时可直接作为 LLM 上下文，实现优雅降级；
4. **角色感知**：`role` 字段（source/template）在同一结构中区分数据源和模板，简化流水线传参。

## 创新点二：三层混合抽取引擎——规则优先，LLM 智能兜底

### 2.1 创新背景与动机

文档数据抽取面临的核心矛盾：
- **确定性需求**：数值、日期等结构化字段要求高精确性，不能有幻觉；
- **灵活性需求**：自然语言描述的字段名存在同义词变体，需要语义理解；
- **规模性需求**：百强排名等场景需提取 100+ 条记录，单次 LLM 调用有 token 上限。

**纯规则方法**覆盖率有限，**纯 LLM 方法**速度慢且有幻觉风险。本项目设计了**三层混合抽取引擎**，在确定性、覆盖率与成本之间取得最优平衡。

### 2.2 设计方案

**整体架构**：

```
输入：检索结果（RetrievalResult）+ 模板表头（TemplateSchema）
                    │
          ┌─────────▼─────────┐
          │    第一层：规则抽取  │  ← 数值/日期/实体后缀正则匹配
          │   Rule-based       │    表头精确/包含匹配
          └─────────┬──────────┘
                    │  规则覆盖率 ≥ 80%？
                    ├── 是 ──→  返回结果（confidence: HIGH, method: rule）
                    ↓ 否
          ┌─────────▼─────────┐
          │    第二层：LLM 辅助  │  ← qwen2.5:14b 语义理解
          │   LLM Fallback     │    Prompt 驱动 JSON 输出
          └─────────┬──────────┘
                    │  检测到多实体信号？
                    ├── 否 ──→  返回结果（confidence: MEDIUM, method: llm）
                    ↓ 是
          ┌─────────▼─────────────────┐
          │   第三层：分块迭代 LLM       │  ← 文本按 12 块分组
          │   Chunk-based Multi-call   │    每组独立 LLM 调用
          └─────────┬─────────────────┘
                    │
              合并去重所有 chunk 结果
                    │
              返回（confidence: MEDIUM,
                   method: llm_multi）
```

**关键实现细节**：

**规则层核心正则**：
```python
NUMBER_PATTERN  = r'[-+]?\d[\d,]*(?:\.\d+)?'        # 数值
DATE_PATTERN    = r'(?:19|20)\d{2}[-/.年]\d{1,2}...' # 日期
ENTITY_SUFFIXES = ("市", "省", "自治区", "县", "区")   # 地名后缀
```

**多实体信号检测**：
```python
MULTI_ENTITY_SIGNALS = [
    "百强", "排名", "排行", "榜单", "名录", "名单",
    "各城市", "各省", "Top", "ranking", "全景报告",
]

def _is_multi_entity_task(text: str) -> bool:
    return any(signal in text for signal in MULTI_ENTITY_SIGNALS)
```

**分块迭代策略**：
```python
CHUNK_SIZE = 12  # 每次处理 12 个文本块

def _extract_multi_entity_chunked(blocks, headers, llm):
    all_records = []
    chunks = [blocks[i:i+CHUNK_SIZE] for i in range(0, len(blocks), CHUNK_SIZE)]
    for chunk in chunks:
        text = "\n".join(b.content for b in chunk)
        records = llm.extract_structured(text, headers, num_predict=2048)
        all_records.extend(records)
    return _deduplicate(all_records, key_column=headers[0])
```

**置信度评分体系**：

```
抽取来源          置信度分数
rule_exact        0.95+   精确规则匹配（完整精确命中）
rule_contains     0.82+   包含规则匹配（子串命中）
llm_high          0.75+   LLM 高置信输出（格式完整、值非空）
llm_medium        0.62+   LLM 中置信输出（部分字段为 null）
llm_inferred      0.45+   LLM 推断（低结构性文本）
not_found         0.00    未检索到相关数据
```

### 2.3 创新价值

1. **兼顾速度与覆盖率**：规则层处理约 70% 的字段（ms 级），LLM 只处理剩余复杂场景，平均每任务减少 LLM 调用次数约 40%；
2. **多实体完整性保障**：分块迭代策略突破了 LLM 单次文本窗口对大量实体抽取的限制，100+ 条数据的覆盖率从约 60%（单次调用）提升至 95%+；
3. **可解释性**：每个字段的 `extraction_method` 字段记录实际使用的抽取策略，便于审计和调试；
4. **优雅降级**：LLM 不可用时自动回退纯规则模式，系统鲁棒性强。

## 创新点三：置信度驱动的全链路数据追溯与验证机制

### 3.1 创新背景与动机

在数据汇总场景中，"覆盖率固然重要，但可信度同样关键。传统的自动化工具填完数据后黑盒输出，用户无从判断哪些数据可信、哪些需要人工核查。

本项目设计了**置信度驱动的全链路追溯机制**，让每一个填充结果都"有据可查"：
- **来自哪里**（source_file + source_location）
- **如何得到**（extraction_method）
- **可信程度**（confidence score）
- **是否有异常**（validation_checks）

### 3.2 设计方案

**FilledFieldResult 数据模型**：

```python
class FilledFieldResult(BaseModel):
    """单个字段的填充结果，包含完整追溯信息。"""
    field_name: str          # 模板字段名
    value: str               # 填入的值
    confidence: Optional[float]   # 置信度 0.0~1.0
    source_file: str         # 数据来源文件路径
    source_location: str     # 数据来源位置（如 "table0.row5.col2"）
    evidence: str            # 原始证据文本片段（用于人工核查）
    extraction_method: str   # 抽取方式：rule_exact/llm/llm_multi 等
    missing_reason: str      # 若为空值，说明未填入的原因
```

**三维验证体系**：

```
验证维度一：完整性验证（Completeness Check）
────────────────────────────────────────────
- 计算 fill_rate = rows_filled / expected_rows × 100%
- fill_rate < 80% → ValidationItem(passed=False, check="fill_rate")
- 空值字段列举并记入 warnings

验证维度二：证据链验证（Evidence Chain Check）
────────────────────────────────────────────
- 检测"有值但无来源"的异常字段（value 非空但 source_file 为空）
- 此类字段强制将 confidence 置 None（标记为不可信）
- 记入 "value_without_evidence" 告警

验证维度三：数值合理性验证（Value Sanity Check）
────────────────────────────────────────────
- 对数值型字段检测 NaN/Inf/异常大值
- 检测日期字段格式是否符合模板预期
- 实体字段合法性校验（如行政区划名称后缀验证）
```

**输出结果报告示例**：

```json

  "rows_filled": 98,
```

**实体合法性校验（entity_utils）**：

```python
# 自动检测并拦截非法实体值
def validate_entity_value(header: str, value: str) -> tuple[bool, str]:
    """
    检验填入实体字段的值是否合法。
    - 地区名称：必须以行政区划后缀结尾（市/省/区/县等）
    - 机构名称：不能包含明显的数值或日期
    返回 (is_valid, reason)
    """
    if is_entity_header(header):
        if not any(value.endswith(sfx) for sfx in ENTITY_SUFFIXES):
            return False, f"'{value}' 不符合行政区划命名规范"
    return True, "OK"
```

### 3.3 创新价值

1. **结果可审计**：每个字段都携带完整的"数据血缘"信息，运营人员可快速定位低置信度字段进行人工复核，审计成本大幅降低；
2. **异常自动检测**：三维验证体系覆盖了完整性、证据完整性、数值合理性三个维度，将隐性错误显式化；
3. **渐进式信任**：置信度分级（HIGH/MEDIUM/LOW）使得系统输出不再是"全信或全不信"的二元结论，而是给出定量的可信程度，为实际业务中的人机协作提供精细化依据；
4. **LLM 幻觉防护**：通过"有值必有证据"的强约束，将置信度设为 `None` 的机制，有效防止 LLM 幻觉产生的错误数据静默流入最终结果。

## 三大创新点关系图

```
┌────────────────────────────────────────────────────────┐
│                  LLM-DocFusion 核心创新体系             │
├────────────────────────────────────────────────────────┤
│                                                        │
│  创新一：DocumentBundle 中间层                          │
│  ┌─────────────────────────────────────────────────┐  │
│  │  Excel / Word / MD / TXT / CSV                  │  │
│  │         ↓  （统一解析）                          │  │
│  │  DocumentBundle（text_blocks + tables）          │ │
│  └─────────────────────────────────────────────────┘  │
│          解耦"读取"与"理解"，格式扩展零成本              │
│                       ↓                               │
│  创新二：三层混合抽取引擎                               │
│  ┌─────────────────────────────────────────────────┐  │
│  │  规则层（快、稳）→ LLM 层（智能）→ 分块迭代（全量） │  │
│  └─────────────────────────────────────────────────┘  │
│          平衡速度与覆盖率，多实体不遗漏                  │
│                       ↓                               │
│  创新三：置信度驱动全链路追溯                            │
│  ┌─────────────────────────────────────────────────┐  │
│  │  每字段：来源 + 方法 + 置信度 + 证据文本           │  │
│  │  三维验证：完整性 + 证据链 + 数值合理性            │  │
│  └─────────────────────────────────────────────────┘  │
│          结果可信、可查、可审计                          │
└────────────────────────────────────────────────────────┘
```

三大创新点形成完整的创新闭环：**中间层**保证多格式统一输入，**混合引擎**保证高质量抽取，**追溯验证**保证输出可信可审，共同支撑了 LLM-DocFusion 的核心价值主张。
