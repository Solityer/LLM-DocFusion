# LLM-DocFusion 团队补充材料

## 一、技术选型深度说明

### 1.1 为什么选择 qwen2.5:14b 而非 GPT-4 / Claude 等云端模型

| 对比维度 | 云端 API（GPT-4/Claude） | qwen2.5:14b（本地） |
|:--------:|:-----------------------:|:------------------:|
| 数据隐私 | 数据传输至云端，存在泄露风险 | **完全本地，零外传** |
| 使用成本 | 按 token 计费，大规模使用成本高 | **一次部署，免费使用** |
| 中文能力 | 较好 | **专门优化，政务/统计文本理解更佳** |
| 网络依赖 | 必须联网 | **离线可用，适合企业内网** |
| 响应延迟 | 依赖网络质量 | **本地推理，延迟稳定** |
| 合规性 | 涉及出境数据合规问题 | **符合数据本地化要求** |
| 可控性 | 模型更新不受控 | **版本固定，行为可预期** |

**结论**：在政务、金融、医疗等对数据隐私要求严格的场景，本地化 LLM 方案是唯一合规且经济的选择。qwen2.5:14b 作为国产中文大模型的代表，在中文文档理解场景中表现优异。

### 1.2 为什么不使用 RAG（检索增强生成）向量检索

当前版本的检索策略采用**关键词规则 + 启发式匹配**，而非向量数据库（FAISS/Chroma 等）。原因：

1. **目标场景特点**：用户明确提供了数据源文件，检索范围有边界，不是开放域 QA；
2. **精确性优先**：统计数据提取需要精确匹配，语义相似但内容不同的段落反而是噪声；
3. **部署复杂度**：避免引入向量数据库依赖，降低部署门槛（一键启动）；
4. **计划方向**：向量检索列入 v2.0 迭代计划，用于处理超大文档集场景。

## 二、系统局限性与改进方向

### 2.1 已知局限性

| 局限性 | 具体表现 | 影响程度 |
|:------:|:--------:|:--------:|
| 不支持 PDF | 无 PDF 解析能力，需预先转换 | 中 |
| 扫描件无法处理 | 图片型文档（扫描 PDF/图片）无法解析 | 中 |
| 模型推理速度 | CPU 模式下单任务约 30～90 秒 | 低（可接受） |
| 超大文档 | 单文件超 50 MB 时性能下降 | 低 |
| 复杂嵌套模板 | 多层次嵌套单元格的 Excel 模板支持有限 | 低 |
| 并发处理 | 当前为单队列处理，不支持高并发 | 低（单机场景） |

### 2.2 后续迭代计划

**v2.0 计划（近期）**：
- [ ] PDF 解析支持（基于 PyMuPDF 或 pdfplumber）
- [ ] OCR 支持（扫描件）
- [ ] 向量检索增强（RAG 升级）
- [ ] 模板字段自动推断（无需标注占位符）

**v3.0 计划（中期）**：
- [ ] Docker 容器化部署
- [ ] 多用户协作与任务历史
- [ ] 更大参数量模型支持（72b）
- [ ] 数据库数据源直连

## 四、代码质量与工程规范

### 4.1 代码组织原则

本项目严格遵循以下工程规范：

- **类型注解**：所有函数均添加类型注解（Python Type Hints），提高代码可读性和 IDE 支持；
- **Pydantic 数据模型**：所有数据结构使用 Pydantic v2 BaseModel 定义，保证类型安全和自动文档生成；
- **单一职责**：每个 service 模块只负责一项具体功能，模块间通过标准接口通信；
- **结构化日志**：使用统一日志模块 `core/logging.py`，关键操作均有 INFO 级日志；
- **异常层次**：定义了专用异常类（`DocumentReadError`, `FillError`等），错误信息语义清晰；
- **配置外化**：所有可调参数集中在 `config/config.json`，不在代码中 hardcode。

## 五、与同类系统的对比分析

### 5.1 对比方案

| 系统/方案 | 类型 | 优势 | 劣势 |
|:--------:|:----:|:----:|:----:|
| Microsoft Power Automate | 商业 RPA | 功能完整，生态好 | 高成本，需专业配置，无 LLM 语义理解 |
| Python 自定义脚本 | 定制开发 | 灵活性高 | 每个模板需重写，无法复用 |
| ChatGPT 手动粘贴 | 手动+AI | 零部署 | 手动操作，数据不安全，不可批量化 |
| LangChain + RAG 方案 | 开源框架 | 语义检索能力强 | 依赖配置复杂，不针对文档填充场景优化 |
| **LLM-DocFusion（本项目）** | **专用系统** | **多格式+LLM+本地+开箱即用** | **PDF/图片暂不支持** |

### 5.2 核心差异化优势

**差异一：专注文档填充场景的端到端系统**
> 现有 LLM 应用大多是通用聊天助手或 RAG 知识库，LLM-DocFusion 是为"从文档中提取数据、填入模板"这一具体业务场景深度优化的垂直系统，提供从文件上传到结果下载的完整闭环。

**差异二：规则+LLM 混合策略降低幻觉风险**
> 纯 LLM 方案在处理精确数值时存在幻觉风险（如 GPS 数值被"记忆"替换）。本项目的规则优先策略将精确数值字段的幻觉率降低到接近零。

**差异三：完全本地化，满足数据安全合规**
> 对比云端 API 方案，本项目实现了业界少有的"多格式支持 + LLM 理解 + 完全本地化"三位一体，是目前（截至 2026 年）该组合下功能最完整的开源实现之一。
